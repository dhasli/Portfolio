Dell EMC NetWorker Data Domain Incident Response 1.7PB War Story

☠️ 1.7 Petabytes. Gone. (Well, Almost.)

A phantom server that "didn't exist" responded to ping. A test environment had quietly touched production storage. And suddenly 1.7 petabytes of backup data was hanging by a thread — protected by one compliance setting that someone had the good sense to enable years ago.

Shibu A · 14 min read · May 2025 · — views
The Setup

☕ Just Another Routine Day at the ODC

The Dell EMC Offshore Development Center inside Wipro had a certain rhythm to it. Tickets came in, engineers caffeinated themselves into functionality, and the day moved along in the comfortable hum of enterprise support work. My colleague and I were doing exactly that — the usual morning routine of checking the queue, triaging cases, pretending the coffee was better than it was.

Then a case arrived that looked, at first glance, completely ordinary:

📋
"Some Data Domain devices are missing or inaccessible from the NetWorker server."

Routine device issue. Probably a mount problem. Definitely resolved before lunch.
That was the last optimistic thought either of us had for a while.

Now — context. Because the scale of this environment matters enormously for understanding why "some devices are inaccessible" was not a small statement.

📐 Customer Environment Scale
NetWorker Servers
15 servers
Max clients/server
~2,000 (some push 4,000)
Affected DD devices
6 devices, 1 DD

"Clients" in NetWorker terminology doesn't mean laptops or workstations — it means actual servers, applications, databases. Infrastructure. So we're talking about a backup environment protecting the kind of data that, if lost, would make very serious people very upset in very rapid succession.

My colleague and I both had a thing for the complicated cases — the ones where the logs don't immediately cooperate and the answer isn't on page one of any documentation. We picked this one up together. We should have made more coffee first.

The Error

🚨 An Error We'd Never Seen Before

We joined the remote session and checked the affected Data Domain devices. Visually, they were all present — listed in NetWorker, no connectivity alarms, nothing obviously broken. Good sign, we thought.

Then we tried to mount one.

NetWorker daemon log output
# Attempting to mount production device...
nsrmmd media warning:
  /ddvar/ddboost/PROD_MTREE_01/...
  expected volume 'Master_Vol_001'
  found 'Test_Vol_001'

Odd. We moved to the next device.

nsrmmd media warning: /ddvar/ddboost/...
expected volume 'Master_Vol_002' found 'Test_Vol_002'

And the next. And the next. Same pattern across six devices on the same Data Domain. The production NetWorker server expected its own volume labels — and found labels belonging to something called Test_Vol.

🧠
What this error actually means: NetWorker tracks every device using a volume label — essentially the device's identity. This error means something had physically rewritten the label on the storage. NetWorker wasn't confused. It was correctly reporting that the storage now belonged to a different server.

We had no internal documentation for this error. I don't think many people had ever hit it before, because hitting it requires a very specific sequence of bad decisions — which we were about to uncover.

Standard procedure: collect logs, ask the backup team about recent changes.

Their response, delivered with complete confidence:

🙃
"Nothing was changed."

Ah yes. The universal customer incantation. Spoken in every language, on every continent, in every enterprise support case since the dawn of IT. Nothing was changed. The system just spontaneously developed a different volume label. Gremlins. Cosmic rays. Mercury in retrograde.

Sure.
The Ghost

👻 The Server That Didn't Exist

We kept digging. Inside the NetWorker daemon logs, one name kept showing up. Buried in the metadata associated with those rogue volume labels:

# Volume ownership metadata extracted from daemon logs
Server origin: Test_Networker
Datazone ID: [ does not match production ]
Volume label: Test_Vol_001 → Test_Vol_006

A NetWorker server called Test_Networker. Its fingerprints were all over those volumes.

We asked the backup team: do you have a server with this hostname?

Their response, once again delivered with great conviction: "No. We don't have any server with that name."

Now here's the thing about NetWorker. It does not invent server names. It doesn't hallucinate metadata. It doesn't wake up one morning, feel creative, and decide to randomly associate production volumes with a fictional server. NetWorker is deterministic. If that server name is in the logs, something with that hostname existed and touched those devices.

We asked them to search internally for the hostname anyway. Not because we expected anything — but because the logs don't lie.

A few minutes passed. Then someone on the bridge said — quietly, like they weren't sure they were reading it right:

C:\> ping Test_Networker
Pinging Test_Networker [10.x.x.x] with 32 bytes of data:
Reply from 10.x.x.x: bytes=32 time=1ms TTL=128
Reply from 10.x.x.x: bytes=32 time=1ms TTL=128
Reply from 10.x.x.x: bytes=32 time=1ms TTL=128
⚠️  The server exists. The server is responding.

The bridge call went very quiet.

Not the productive quiet of people thinking. The specific quiet of people whose internal monologue has just gone completely blank. Their team members — the same people who had firmly stated there was no such server — were staring at a ping response on their own screens.

We logged in immediately.

The Crime Scene

🕳️ An Empty Server and a Disappearing Act

The server was real. But when we logged in and looked around — it was almost completely empty. No devices configured. No obvious backup activity. No smoking gun sitting on the desktop labelled "Evidence."

Whoever had used this server had deleted the devices afterward. Probably thought that was enough to cover their tracks.

💡
The mistake they made: Deleting a device configuration from NetWorker removes it from the active management view. It does not erase the audit trail of what that device did while it was configured. The microservices running inside NetWorker log everything — and those logs don't care about your cleanup job.

We extracted audit logs from every service that had been running inside that test server and brought everything back to our lab. This was no longer support work. This was digital forensics.

🔌
Step 1 — The Connection
Production DD devices added to Test_Networker
Someone connected the production Data Domain storage units — belonging to the master NetWorker server — into the test environment. The test server cheerfully accepted them.
🏷️
Step 2 — The Label
Label operation executed from test server
A label command was run against the devices. NetWorker wrote the test server's volume IDs and datazone metadata directly onto the production storage. The original production labels were overwritten.
🗑️
Step 3 — The Cover-Up
Devices deleted from test server
After the operation, the devices were removed from the test server's configuration. Presumably on the theory that if the devices were gone, there was nothing to explain. The logs, unfortunately, had other ideas.
💥
Step 4 — Discovery
Production server can no longer read its own volumes
The production NetWorker master server expected its own labels. Found labels belonging to a server it had never heard of. Six devices, one Data Domain, completely inaccessible.

The label operation is what caused everything. Here's what it actually does under the hood — because this is the part most people don't realise:

Before label operation
Volume IDs belong to Master NetWorker
Production datazone UUID in metadata
Master server can read, mount, restore
1.7PB of backup data — fully accessible
⚡
LABEL
CMD
After label operation
Volume IDs now belong to Test_Networker
Test datazone UUID overwrites metadata
Master server says: "That's not mine"
1.7PB — technically present, functionally unreachable
The Number

🔢 The Calculation Nobody Wanted to Finish

Root cause confirmed. The next question was the one that makes backup engineers' stomachs drop:

How much data was affected?

We had the media database and backup indexes from the master server — intact, thankfully. We filtered, we counted, we cross-referenced. The calculation wasn't complicated. The answer, however, was.

1.7 PB
of production backup data — inaccessible
That's 1,700 terabytes. 1,700,000 gigabytes. Roughly 340 million copies of a 5GB file.

I remember staring at the screen for a second longer than was professional. The kind of pause where your brain is absolutely certain it has made an arithmetic error and is politely waiting for someone to correct it.

No one corrected it.

The entire bridge call went silent. Not the dramatic movie kind of silence with swelling strings. The real kind — where experienced engineers stop talking because they're all simultaneously calculating consequences in their head and none of those calculations are producing encouraging results.

We escalated. Immediately. Management joined. SMEs were called in. War-room mode activated. And we started preparing the most stressful timeline email I'd ever had to help draft.

The Lifeline

🔒 The One Setting That Saved Everything

Here's where luck — or rather, whoever had the foresight to enforce compliance policies — saved the entire situation.

Data Domain Storage — Layer Analysis
🏷️
Volume Label Layer
NetWorker metadata — volume IDs, datazone UUIDs, ownership information. This layer was overwritten by the test server's label operation.
Compromised
📁
Directory Structure Layer
Volume header files, control folders, sequence counters. Partially affected — the test server modified root-level control files.
Partially affected
🔒
Backup Data Chunks — Retention Lock Active
The actual backup payload. When the test server attempted to delete these files via DD Boost, the Data Domain OS intercepted every request and replied: Permission Denied. Object Locked. Not a single chunk was deleted.
Fully intact

DD Retention Lock (Compliance mode) is the kind of feature that seems bureaucratic and over-cautious right up until the day it's the only thing standing between your organisation and a catastrophic recovery failure. The test server's nsrim process tried to purge everything. The Data Domain said no. Firmly. Repeatedly. Without exception.

The data was physically there. The labels were wrong. Those are two very different problems, and only one of them is recoverable.

The Debate

⚔️ The Recovery Argument (And Why the SME Won)

With the SME team on the call, we started discussing recovery options. My colleague and I had an idea that seemed logical on the surface:

"The test server broke the labels by relabelling. So reconnect to the master and relabel back. Simple."

The SME said no. Immediately. Without hesitation.

❌
Our proposal
Reconnect → Relabel back to Master → Restore
If the problem was a wrong label, just write the correct label back. The master server owns it, so label it as master. Logical, right?

Wrong. The label command doesn't just write a name. It wipes the active directory tree — the volhdr file, internal control folders, sequence counters — and resets everything to zero. You'd corrupt the outer metadata layer a second time, and this time with the retention lock's owner at the controls. nsrim -X would then kick in, see a freshly labelled volume with no matching media database records, and start queuing the data for deletion the moment the 30-day lock expired.
✅
SME approach
Reconnect → Read-Only → Scan → Rebuild → Restore
Treat the storage as a read-only raw dump. Don't touch the label. Don't write anything. Use scanner -m -i to read the raw data chunks directly, extract their metadata headers, and inject everything back into the master server's media database. The scanner is specifically designed to do this without modifying the volume state — which means nsrim never gets triggered, nothing gets queued for deletion, and the data chain stays intact.
🔧 What scanner -m -i actually does
-m flag

Forces the server to extract raw metadata headers from the physical backup files (save sets) sitting on the Data Domain and rebuild the media database entries — without touching the volume label or triggering any write operations.

-i flag

Reconstructs the Client File Indexes (CFI) — the file-level directory trees that allow browsable, file-by-file restores. Without this, you can recover save sets but can't browse individual files.

Why it worked

The test server only corrupted the label layer. The physical backup chunks beneath it were locked and untouched. Scanner skipped the corrupted root label entirely, read the locked archive chunks directly, and re-registered everything back into the master's database.

The Resolution

✅ The Restore That Changed the Mood

🛠️ Recovery Sequence — Executed by SME Team
1
Reconnect all 6 affected devices to the production master NetWorker server
2
Mark all volumes as read-only — prevent any write operations or nsrim triggers
3
Set scan-needed flag on every affected device
4
Execute scanner -m -i against all 6 devices — rebuild media database and client file indexes
5
Perform test restore — validate data accessibility end-to-end
6
Create new devices in the same DD / same Mtree for ongoing backup operations — affected devices preserved as recovery-only

Hours of scanner output. Watching metadata rebuild entry by entry. And then — the test restore.

✅
Test restore: succeeded.

That single result changed the entire atmosphere of the bridge call. The silence that had settled in when we announced 1.7PB evaporated. People started talking again. Some of them probably started breathing again.

We handed the forensic findings — the complete audit log trail showing exactly what happened and when — to the customer's management team. What they did with it, and what happened to whoever ran those label operations, we never found out. That wasn't our problem to solve. Our problem was the data. The data was back.

Takeaways

🧠 What This Case Permanently Changed

There's a Dell EMC KB article that documents this exact scenario — sharing Data Domain volumes across multiple NetWorker datazones. It's been there for years. It exists because enough people made this mistake that someone had to write it down officially:

📖
Dell KB 000224754: "DD Volume sharing across multiple NetWorker Datazones can cause data loss"
→ Read the official documentation
01
Test environments need real isolation
A test server connected to production storage isn't a test environment. It's a production environment with worse change control and a different name.
02
Deleting config ≠ deleting evidence
Removing a device from NetWorker's GUI removes it from the active view. It does nothing to the audit logs, daemon logs, or service history. Everything is still there. We will find it.
03
Retention Lock is not optional
Without compliance retention lock enabled on that Mtree, the test server's nsrim process would have permanently deleted 1.7PB. No recovery. No scanner trick. No happy ending. Enable it.
04
"Nothing was changed" is almost never true
Infrastructure doesn't behave randomly. If something broke, a human action caused it — sometimes days or weeks before the symptoms appeared. The logs will tell you what it was.
05
Never relabel a compromised volume
The instinct to "fix it the same way it broke" is wrong here. Label commands modify volume state and trigger background processes. In a recovery scenario, read-only and scanner is the only safe path.
06
Scale makes everything worse
At lab scale, a label mistake is a learning experience. At 1.7PB, it's a career-defining incident. The principles are the same. The consequences scale with the environment.

NetWorker isn't a chaos monkey. It isn't a drunken server that randomly starts overwriting things. It does exactly what humans tell it to do — and sometimes humans tell it to do genuinely terrible things. The system executed perfectly. The input was the problem.

That's the uncomfortable truth about enterprise backup incidents. The tools almost never fail. The processes around them do.
